API summary – names and intended use
====================================

1) data-pipeline-service (port 8082)
------------------------------------

- GET /api/portfolio/{customerId}
  - Intended use: Return the stitched insurance portfolio for a specific customer from the unified_portfolio collection (all matched policies across insurers).

- GET /api/advisory/{customerId}
  - Intended use: Return coverage advisory / gap analysis for a customer, based on their stitched portfolio (recommendations and notes).

- POST /api/pipeline/run
  - Intended use: Trigger the full ingestion and stitching pipeline over existing insurer collections already in MongoDB (standardize + stitch into unified_portfolio).

- POST /api/pipeline/upload
  - Intended use: Accept a CSV file (multipart form-data: file + collectionName), ingest it into the correct MongoDB collection, then standardize and stitch policies to customers.

- GET /insurer-portal
  - Intended use: Serve a simple HTML page for insurer users to upload their policy CSV files.

- GET /actuator/health
  - Intended use: Basic health check endpoint for data-pipeline-service (used by monitoring or verify-services script).


2) policy-service (port 8085)
-----------------------------

- POST /api/v1/policies
  - Intended use: Create a new policy record for a customer with all relevant details (policyNumber, insurerId, premium, sumAssured, dates, etc.).

- GET /api/v1/policies
  - Intended use: Retrieve a list of all policies in the system (admin or reporting scenarios).

- GET /api/v1/policies/{id}
  - Intended use: Fetch a single policy by its unique ID (view or manage one policy).

- GET /api/v1/policies/customer/{customerId}
  - Intended use: Fetch all policies that belong to a specific customerId.

- PATCH /api/v1/policies/{id}/status?status=...
  - Intended use: Update the status of a policy (for example, ACTIVE, LAPSED, CANCELLED).

- DELETE /api/v1/policies/{id}
  - Intended use: Delete a policy record by ID.

- GET /api/v1/health
  - Intended use: Health check endpoint for policy-service.

- GET /api/v1/ping
  - Intended use: Lightweight ping endpoint to verify that policy-service is reachable.


3) Integration of backend services via BFF (bff-service, port 8090)
-------------------------------------------------------------------

- POST /api/bff/auth/login
  - Intended use: Frontend login using Full Name + PAN; BFF talks to customer-service, verifies credentials, and returns a JWT token plus customer summary.

- GET /api/bff/portfolio/{customerId}
  - Intended use: Provide the frontend with a unified portfolio for a customer (customer details + aggregated policies + total premium and coverage), combining data from customer-service and data-pipeline-service.

- GET /api/bff/advisory/{customerId}
  - Intended use: Provide the frontend with coverage advisory for a customer, proxied from data-pipeline-service’s advisory API.

- GET /api/bff/insights/{customerId}
  - Intended use: Return additional coverage insights and recommendations, computed in the BFF using portfolio data.

- POST /api/bff/upload
  - Intended use: Allow the frontend (or an operations UI) to upload insurer CSV files (multipart form-data: file, uploadedBy, insurerId), which the BFF then forwards into the ingestion pipeline.

- GET /api/bff/upload/status/{jobId}
  - Intended use: Check the ingestion job status for a previously uploaded file.

- GET /api/bff/health
  - Intended use: Health check endpoint for the BFF (for monitoring and smoke tests).

- GET /api/bff/ping
  - Intended use: Simple connectivity/ping endpoint to verify the BFF is up.


4) Connection from frontend to backend (through BFF)
----------------------------------------------------

- POST /api/bff/auth/login
  - Intended use (frontend): User submits Customer ID/User ID and password (PAN) from the Flutter login screen; receives JWT + customerId for future calls.

- GET /api/bff/portfolio/{customerId}
  - Intended use (frontend): Dashboard screen loads the customer’s full insurance portfolio using the JWT token.

- GET /api/bff/advisory/{customerId}
  - Intended use (frontend): Advisory or recommendations panel shows coverage gaps and suggestions based on the advisory response from BFF.

- POST /api/bff/upload (optional, if exposed in UI)
  - Intended use (frontend/admin): Operations user uploads new insurer datasets, which flow into the data-pipeline-service via BFF.
